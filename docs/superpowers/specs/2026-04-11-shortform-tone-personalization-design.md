# 숏폼 대본 톤 개인화 + 공감 루프 구조 — 디자인

- 작성일: 2026-04-11
- 영향 도구: shortform.html, blog-writer.html, api/shortform-script.js
- 관련 메모: project_session_0411.md (4/11 ElevenLabs 푸시 직후 후속 작업)

## 배경

### 문제 1 — Point/CTA "나열 정리체"

현재 숏폼 대본 생성 결과는 후킹(Hook) 섹션은 잘 작동하지만 본문(Point)과 마무리(CTA)가 "나열 정리체" 느낌이 강하다. 운영자 피드백:

> 본문과 CTA만 조금 더 사람 느낌으로 경험의 예시를 담으면 충분할 것 같아. 현재는 그냥 나열 정리만 하는 느낌이라서.

`api/shortform-script.js` 현재 프롬프트 분석:

- **Point 섹션 (line 75)**: `"나열 금지, 구체적 사례와 숫자로 깊이 있게"` — 추상적이라 AI가 결국 나열로 회귀
- **CTA 섹션 (line 76-81)**: 4종 정형 문구(`"궁금한 점 댓글로 남겨주세요"`, `"유용했다면 팔로우 부탁드려요"` 등) 중 선택. 정확히 "정리체"의 원흉

### 문제 2 — 시청자 공감 부족

운영자가 fingr.io 롱폼 대본을 분석한 결과, fingr가 잘하는 것은 **"공감될만한 질문 → 이유/해법" 루프를 4~5회 반복**하는 것이다. 도입부 1/3 지점까지 시청자가 "내 이야긴가?" 느끼게 만들어 끝까지 볼 동기를 부여한다. 우리 숏폼은 후킹 1번 외에는 공감 장치가 없다.

### 문제 3 — 채널 생존 (다양성 신호)

YouTube Partner Program이 2025년 7월 "**mass-produced inauthentic content**" 정책을 강화했다. AI 사용 자체가 아니라 "**같은 템플릿/구조/목소리/비주얼 반복**"이 채널 비활성화/수익화 박탈로 이어진다. 사람의 개입을 보여주는 다양성 신호가 핵심 해자다.

다양성 신호 매트릭스:

| 신호 | 우리 도구 대응 | 강도 |
|---|---|---|
| Hook 다양성 | 6종 공식 (현재) | ⭐⭐⭐ |
| **개인 경험** | personaMemo (신규) | ⭐⭐⭐⭐⭐ |
| 음성 다양성 | Supertone 6종 + ElevenLabs 6종 | ⭐⭐⭐ |
| 비주얼 다양성 | B-roll FLUX/Imagen/Grok + 5 컨셉 | ⭐⭐⭐ |
| 구조 다양성 | 길이별 5/7/9씬 + 변주 | ⭐⭐ |

memo가 별 5개인 이유: AI가 만들 수 없는 1인칭 경험은 ML이 흉내낼 수 없는 fingerprint다. 이게 도구의 진짜 해자.

### 패턴 참고

블로그 글 작성기(blog-writer.html)는 이미 `#memo` 필드로 운영자 페르소나/경험을 받아 1인칭 톤으로 가공하는 검증된 패턴을 갖고 있다. 같은 패턴을 숏폼에 이식한다.

## 목표

1. 운영자의 페르소나/경험을 메모로 받아 AI가 숏폼 톤(28자/씬)으로 압축해 본문에 자연스럽게 녹임
2. fingr식 "공감 질문 → 마음읽기 → 상황 → 해법 → CTA" 루프 구조를 길이별 변주로 적용
3. 정형 CTA 폐기, 동료 호출형 톤으로 통일
4. 다양성을 유지해 채널 생존 신호 확보 (Hook 6종 유지)

## 입력 경로

두 가지 진입 경로가 있다. 두 경로 모두 `api/shortform-script.js` request body의 신규 필드 `personaMemo`로 통합 전송된다. AI는 메모 길이와 무관하게 동일한 방식으로 가공한다.

### 경로 A: blog-writer 경유

기존 `blogTextForImagePro` 핸드오프 패턴을 복제한다.

- **송신**: blog-writer.html에서 "숏폼 만들기" 버튼 클릭 시 `localStorage.setItem('blogTextForShortform', JSON.stringify({body, memo}))` 후 shortform.html로 이동
- **수신**: shortform.html `DOMContentLoaded`에서 키 존재 여부 확인 → 있으면 `topic`/`blogText`/`personaMemo` 필드에 자동 채움 → 키 삭제(1회성)
- 메모는 블로그 작성기의 긴 단락 그대로 (수십 줄 가능)

### 경로 B: 직접 진입

shortform.html에 신규 optional textarea 추가.

- **위치**: 기존 topic 입력 필드 아래
- **레이블**: "내 경험 한 줄 (선택)"
- **placeholder**: "본인의 경험·페르소나·실패담을 한 줄로 적어주세요. 비워도 됩니다.\n예: 15년차 헤어 디자이너, 처음엔 반신반의했는데 한 달 뒤 머릿결이 확 달라짐"
- **저장 동작**: 마지막 입력 자동 저장(`localStorage.shortformLastMemo`) → 다음 진입 시 자동 채움
- **검증**: 길이 제한 없음, optional

## 대본 구조 (길이별 변주)

### 30초 / 5씬 — 1사이클

```
0. 공감 질문         "요즘 ~ 힘드시죠?"                       (Hook)
1. 마음읽기 질문      "솔직히 ~ 싶으셨을 거예요"                (Point 시작)
2. 상황 설명/통념     "그런데 사실 ~"                          (Point)
3. 해법 + 경험        "저도 ~했었어요" / 메모 녹임              (Point)
4. CTA 동료 호출      "비슷한 경험 있으면 댓글로"               (CTA)
```

### 45초 / 7씬 — 1.5사이클

```
0. 공감 질문                                                  (Hook)
1. 마음읽기 질문                                              (Point)
2. 상황 설명                                                  (Point)
3. 통념 깨기 (변주)                                           (Point)
4. 약속/핵심 해법                                             (Point)
5. 구체 사례 — 메모 녹임 지점                                 (Point)
6. CTA 동료 호출                                              (CTA)
```

### 60초 / 9씬, 90초 / 12씬 — 2사이클+

본문(scene 2~7)에 다음 변주 블록을 자유롭게 추가:
- 추가 마음읽기 질문
- 두 번째 사례
- 또 다른 통념 깨기
- 작은 약속

**구조 자체의 템플릿화 방지 룰** (프롬프트에 명시):
- 위 구조는 **default**이며, 토픽·메모에 따라 순서 바꾸거나 1단계 생략 가능
- 같은 토픽에 두 번 생성해도 같은 구조가 나오면 안 됨 — Hook 종류, 마음읽기 위치, 사례 위치를 변주

## 프롬프트 변경 (`api/shortform-script.js`)

### Hook 공식 6종 재해석 (line 56-72 수정)

기존 6종을 유지하되 **공감 베이스로 재해석**한다. 다양성 신호를 위해 6종 모두 사용 가능하지만, 시청자 마음을 먼저 얻는 게 출발점이라는 원칙 명시:

```
[후킹 공식 — 6종 다양성 유지, 공감 베이스]
모든 후킹은 시청자가 "내 얘긴가?" 느끼게 만드는 게 출발점.
6종 중 토픽에 가장 잘 맞는 것을 매번 다르게 선택해 다양성 확보.

1. 질문형 (공감 질문 default): "요즘 ~ 많이 힘드시죠?"
2. 충격형: 공감 베이스 통념 뒤집기 → "매출 줄어드는 거, 단순 경기 탓이 아닙니다"
3. 비밀형: "~인 사장님들만 아는 전략이 있어요"
4. 증거형: "이것 하나 바꿨더니 한 달 만에 ~"
5. 공감형: "저도 ~로 6개월 헤맸어요"
6. 경고형: "이 실수 하나가 ~을 다 날립니다"

★ 6종을 매번 다른 것으로 선택할 것. 같은 운영자가 5개 영상 만들면 5개가 다 달라야 함.
★ 정형 출발 금지: "안녕하세요", "오늘은 ~에 대해 알아볼게요", "여러분 ~"
```

### 구조 강제 (line 73-82 교체)

```
[대본 구조 — fingr식 공감 루프]

총 씬 수에 따라 다음 구조를 base로 삼되, 변주 허용:

▶ 5씬 (30초): 공감질문 → 마음읽기 → 상황 → 해법+경험 → CTA
▶ 7씬 (45초): 공감질문 → 마음읽기 → 상황 → 통념깨기 → 약속 → 사례+경험 → CTA
▶ 9씬+ (60~90초): 7씬 base에 [추가 마음읽기 / 두 번째 사례 / 작은 약속] 자유 삽입

★ 핵심: 시청자가 끝까지 보게 하려면 도입부 1/3까지 "내 이야긴가?" 느끼게 만들 것.
   scene[1]에 반드시 "마음읽기 질문" 한 번 더 던질 것 (예: "솔직히 ~ 싶으셨을 거예요")
★ 구조 변주: 같은 base여도 매번 표현·순서가 달라야 함. 템플릿화 금지.
```

### Point 섹션 (line 75 교체)

```
- Point 씬: 추상 나열 금지. 구체 장면/숫자/감정만.

  ★ 메모(personaMemo)가 있으면: Point 중 최소 1씬은 메모를 28자 압축 가공.
    원문 그대로 X — 핵심 감정/장면/숫자만 추출해 숏폼 톤으로 재구성.
    예) 메모 "15년차 헤어 디자이너, 손님이 '여기 물 맛있다'고 했을 때 뿌듯했음"
        → 씬 "손님이 '물 맛있다' 한마디에 울컥했어요"

  ★ 메모가 없으면: 관찰형 1인칭 허용 — "~인 분들 많으세요", "저도 처음엔 ~인 줄 알았는데"

  ★ 나열 vs 경험 예시 (반드시 후자):
    ❌ "키워드 분석, 카테고리 설정, 콘텐츠 발행이 중요합니다"
    ⭕ "어제 알림 50개 떴어요"
    ❌ "꾸준한 노력이 필요합니다"
    ⭕ "6개월간 매일 쓰다가 한 달 쉬었더니 다 무너졌어요"
```

## 안전망

운영자가 결과 톤이 마음에 안 들면 **shortform.html의 기존 자막/대본 편집 기능으로 직접 수정 가능**. 톤·구조를 공격적으로 가도 후수정으로 회복 가능하므로 정형 폴백을 두지 않는다.

## Out of Scope (의도적 제외)

- 자막 편집기 UI 확장 (폰트/색상/외곽선 등 fingr 벤치마크 기능) — 별도 작업
- blog-writer 측 "숏폼 만들기" 버튼 정확한 배치 디테일 — 구현 단계에서 결정
- 메모 내용 검증/필터링 (욕설, 광고 등) — 1차에선 운영자 신뢰
- 페르소나 vs 경험 분리 (페르소나 = 영구 저장, 경험 = 1회성) — 단일 필드 + 자동 저장으로 통합
- 구조 변주를 강제 검증하는 후처리 코드 — 1차는 프롬프트 강제만

## 리스크

1. **AI 가공 정확도**: 메모 압축 시 운영자 의도와 다른 사례 만들 수 있음 → 후수정 흡수
2. **메모 없는 케이스 어색함**: 보수적 톤을 원하는 일부 사용자에겐 동료 호출형 CTA가 어색 가능 → 후수정 가능
3. **localStorage 키 충돌**: `blogTextForShortform`이 기존 키와 겹치지 않는지 확인 (현재 grep으로 미존재 확인됨)
4. **개인정보**: 메모에 운영자 본명/사업명 들어갈 수 있음 → localStorage 한정, 서버 영구 저장 없음
5. **구조 템플릿화**: base 구조가 너무 강제되면 모든 영상이 똑같아져 다양성 신호 손상 → 프롬프트에 변주 룰 명시로 완화. 1차 검증 시 운영자가 같은 토픽 3회 생성해 차이 확인
6. **후킹 다양성 손실**: 6종 중 "공감 질문 default"로 갔다가 매번 1번만 쓰이면 다양성 잃음 → "매번 다르게 선택" 강제 명시

## 검증 방법

1. **수동 비교**: 동일 토픽으로 (메모 있음 / 메모 없음) 각 1회 생성 → 톤 차이 운영자 육안 확인
2. **CTA 정형 문구 금지 검증**: 생성 결과에 `"궁금한 점 댓글로"`, `"감사합니다"` 등 폐기 문구가 0회 등장하는지
3. **핸드오프 E2E**: blog-writer에서 글 작성 → 숏폼 만들기 클릭 → shortform.html에서 자동 채워졌는지
4. **구조 변주 검증**: 같은 토픽 3회 생성 → Hook 종류/마음읽기 위치/사례 위치가 매번 달라야 함
5. **마음읽기 강제 검증**: scene[1]이 항상 질문형 또는 "~싶으셨을 거예요" 류 문장인지
