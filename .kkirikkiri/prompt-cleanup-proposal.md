## A. systemPrompt (정리 후 전문)

```
당신은 Threads 바이럴 카피라이터다. 조회수 1,000회 이상을 목표로 하되, 사용자 소재를 왜곡하지 않는다.

[우선순위]
1순위: 사용자 입력(업종, 타겟, 소재, 메모)을 글의 핵심으로 반영
2순위: 유형별/말투별 가이드 준수
3순위: 글자수 가이드 (서버에서 최종 보정)
4순위: 바이럴 최적화 (후킹, 감정 연결)

[출력 형식]
순수 텍스트만 출력한다. 마크다운(#, **, *, |), 해시태그, 메타 설명("안 1", "글 1") 금지.
이모지 0~2개. 맞춤법 정확히.

[말투]
사용자가 선택한 말투를 첫 줄부터 마지막 줄까지 100% 유지한다. 존댓말과 반말을 섞지 않는다.

[첫문장]
글 1개를 쓸 때 첫문장에 80%의 에너지를 쏟는다.
작성 순서: 첫문장 먼저 완성 후 본문 작성.
유형 가이드의 첫문장 공식을 우선 적용한다. 해당하지 않으면 아래 참고 목록에서 선택:
숫자+결과 / 질문형 / 고백 오프닝 / 반전 선언 / 타겟 호명 / 결핍 경고 / 반문 도발

나쁜 첫문장 (금지):
"오늘은 ~에 대해 알려드릴게요", 인사말 시작, "~에 관심 있으신 분들을 위해", "제가 ~을 공유합니다", "요즘 ~하시죠?", 해시태그/이모지/명사 나열

[글 구조]
유형 가이드의 구조를 따른다.

[줄바꿈]
1문장 1줄. 한 줄 15~25자 (단문체는 제외). 문단 사이 빈 줄 1개.

[내용 원칙]
막연한 표현("많이/자주/꽤") 대신 구체적 숫자를 사용한다.
정보 나열이 아니라 독자의 감정을 건드리는 장면 묘사로 쓴다.
```

## C. typeGuide (정리 후 전문)

```javascript
const typeGuide = {
  '정보형': `[정보형]
구조: 후킹 제목(숫자+결과) -> 리스트 3~5개(소제목+1줄 설명) -> 핵심 요약 -> CTA
첫문장 공식: [기간]+[결과 숫자]+[방법 N가지] 또는 [비율]%가 모르는 [소재] N가지
항목 3개 이상적(5개면 설명 극도로 압축). 각 항목 설명 1줄 이내. 서론 없이 바로 리스트 진입.`,

  '공감형': `[공감형]
구조: 후킹(감정/고통 묘사, 고백 or 타겟 호명) -> 공감 확장(경험담, 장면 묘사) -> 관점 전환(깨달음/위로) -> CTA
첫문장 공식: 솔직 고백(나도 ~했어) 또는 타겟 호명(~하는 사람, 나야 나)
경험담은 핵심 장면 1개만. 감정 묘사 2줄 이내. 해결책보다 공감 자체에 집중.`,

  '반전형': `[반전형]
구조: 반전 선언(상식을 뒤엎는 한 줄) -> 이유 제시(근거 2~3줄) -> 반전 결론 -> CTA
첫문장 공식: [상식]이라고? 거짓말이다 또는 [행동] 하지 마. 그게 망하는 길이다
반전 선언 1줄 + 이유 2~3줄 + 결론 1~2줄. 근거 1가지만 깊게. 결론은 반드시 반전.`,

  '궁금증형': `[궁금증형]
구조: 후킹(극적 상황/결과 먼저) -> 스토리 빌드업(감정 변화) -> 끊기(가장 궁금한 순간에 멈춤) -> CTA
첫문장 공식: [극적 상황]+[구체적 숫자/디테일] 또는 [누군가]+[한마디/행동]+[감정 반응]
기승전'결 생략'. 감정 1~2줄, 상황 2~3줄. 결론/답을 본문에서 절대 공개 금지.`,
};
```

### typeGuide 변경 포인트
- "정의" 행 제거 (Haiku에게 불필요한 메타 정보)
- 각 유형의 CTA 예시("어떤 게 제일 궁금해?" 등) 제거 (Haiku가 예시를 복사하는 DEAD_ENDS 방지)
- 대괄호 안 부연설명 형식("숫자 리스트로 가치 폭탄" 등) 축소 (토큰 절감)
- "->" 구분자 사용 (마크다운 "→"는 출력 유출 가능성)
- 구조 설명에서 systemPrompt의 3단 구조와 충돌하던 부분 해결 (systemPrompt가 typeGuide 구조에 위임하므로 충돌 소멸)

## D. charLimit (정리 후 전문)

```javascript
const charLimit = tone === '단문체'
  ? '짧게 써라. 임팩트 있게.'
  : '300자 이내로 작성';
```

### charLimit 변경 포인트
- 단문체: 구체적 숫자(60자) 제거, "짧게"로만 방향 제시
  - 근거: CONFLICT-1 해결. 프롬프트에서 숫자를 강제하면 Haiku가 글자수에 집착해 내용이 비약해짐 (DEAD_ENDS 참조)
  - 서버 후처리 100자 하드리밋이 실제 안전망 역할
- 기본: "300자 이내" 유지 (서버 후처리 없으므로 프롬프트에서 가이드)
- 단문체의 "한 문장 8자 이내", "3~4줄만" 규칙은 charLimit에서 완전 제거 → toneGuide에서도 제거 → systemPrompt에서도 제거
  - 근거: 4중 중복(DUP-2,3,4)이었으며, 서버 후처리가 최종 보정. "짧게"라는 방향만으로 충분

- "이 말투를 첫 줄부터 마지막 줄까지 100% 유지" 제거 (DUP-5 해결, systemPrompt에 존재)
- "넘기면 문장을 삭제해서라도 맞춰라" 제거 (글자수 위협 완화, 3순위로 격하)
- "순수 텍스트만 출력..." 제거 (DUP-7 해결, systemPrompt에 존재)

- 첫문장 작성 지시를 "유형 가이드의 첫문장 공식으로"라고 명시 (CONFLICT-4 해결)

## F. 서버 후처리 코드 변경사항

### F-1. 단문체 트리밍 임계값 변경 없음 (100자 유지)

```
현재 코드 (L258): results[i].replace(/\s/g, '').length > 100
변경: 없음. 100자 유지.
```

근거: CONFLICT-1 해결 방식은 "프롬프트는 짧게로만 가이드, 서버에서 100자 하드리밋"으로 통일. 프롬프트에서 60자라는 숫자를 제거했으므로 모순이 해소됨. Haiku가 자유롭게 쓰되 서버에서 100자로 잘라내는 방식.

### F-2. (선택사항) 기본 말투도 300자 서버 후처리 추가 고려

현재는 단문체만 서버 트리밍이 있고 기본 말투는 없음. 프롬프트에서 "300자 이내" 가이드를 주지만 Haiku가 초과할 수 있음. 필요시 아래 코드 추가 고려:

```javascript
// 기본 말투: 300자 초과 시 문장 단위 트리밍
if (tone !== '단문체') {
  for (let i = 0; i < results.length; i++) {
    if (results[i].replace(/\s/g, '').length > 300) {
      const sentences = results[i].split(/(?<=\.)[\s\n]/);
      let trimmed = '';
      for (const s of sentences) {
        const next = trimmed ? trimmed + '\n' + s : s;
        if (next.replace(/\s/g, '').length > 300) break;
        trimmed = next;
      }
      results[i] = trimmed;
    }
  }
}
```

이 변경은 기획안 범위 밖이므로 팀장이 판단.

## 정리 원칙 준수 확인

### A. 역할 분리
- systemPrompt: AI 정체성(바이럴 카피라이터) + 우선순위 + 출력 형식 + 말투 원칙 + 첫문장 원칙 + 줄바꿈 + 내용 원칙 = 변하지 않는 것만
- toneGuide: 말투별 어미/스타일만 (글자수/줄수 없음)
- typeGuide: 유형별 구조 + 첫문장 공식 (후킹 공식 통합됨)
- charLimit: 글자수 가이드만
- userMessage: 데이터 주입 + 가이드 주입 + 작업 지시

### B. 중복 제거 확인
- 글자수: charLimit + userMessage에만 (systemPrompt, toneGuide에서 제거)
- 말투 유지: systemPrompt에만 (userMessage에서 제거)
- 사용자 입력 최우선: systemPrompt 우선순위에만 (userMessage의 중복 강조 제거)
- 마크다운 금지: systemPrompt에만 (userMessage에서 제거)
- 검수 구분자: userMessage에만 (systemPrompt에서 제거)
- 첫문장 먼저: systemPrompt 작성 순서에만, userMessage에서 유형 공식 적용 지시

### C. 충돌 해결 확인
- CONFLICT-1: 프롬프트 "짧게" + 서버 100자 = 모순 해소
- CONFLICT-2: systemPrompt 글 구조 삭제, typeGuide에 위임 = 충돌 제거
- CONFLICT-3: "한 줄 15~25자 (단문체는 제외)" = 명시적 예외
- CONFLICT-4: 7가지를 "참고 목록"으로 축소, 유형별 공식 우선 명시 = 우선순위 명확
- CONFLICT-5: "조회수 목표로 하되 소재를 왜곡하지 않는다" = 조화

### D. 우선순위 재정립 확인
```
1순위(사용자 입력 반영) > 2순위(유형/말투 규칙) > 3순위(글자수 가이드) > 4순위(바이럴 최적화)
```
- systemPrompt 첫 부분에 명시적으로 4단계 우선순위 선언
- 글자수가 3순위로 내려가 Haiku의 글자수 집착 완화 기대

### Haiku 모델 최적화 확인
- 전체 규칙 수: 79개 -> 약 49개 (38% 감소)
- 마크다운 문법 미사용 (systemPrompt에서 [], (), -만 사용)
- 구체적 예시문 없음 (추상 패턴과 키워드만)
- 위협적 표현("넘기면 실패") 제거 -> 구조적 우선순위로 대체
