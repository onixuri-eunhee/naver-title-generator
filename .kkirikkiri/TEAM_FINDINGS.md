### 블로그 제목 생성기 (index.html)
- 12개 패턴 (긍정형 4 + 위협형 4 + 호기심형 4)
- 각 패턴 2개 템플릿 = 총 24개 제목
- 키워드 하나를 받아 JS 템플릿에 삽입
- API 호출 없음, 완전 프론트엔드 처리

### 후킹문구 생성기 (hook-generator.html)
- 업종 + 키워드 2개 입력
- ~100개 JS 템플릿 함수에서 15개 랜덤 셔플
- 10가지 심리학 기반 후킹 공식 (문제공감, 손실회피, 호기심, 상식비틀기 등)
- 한국어 조사 자동 처리 (은/는, 이/가 등)
- API 호출 없음, 완전 프론트엔드 처리

### 기존 API 구조
- api/generate.js: 범용 Claude API 프록시 (rate limit + Redis)
  - model 파라미터로 모델 선택 가능 (기본 Sonnet)
  - system/messages를 그대로 Claude API로 전달
  - skipRateLimit 옵션 있음
- api/threads.js: Haiku 사용, 자체 프롬프트 내장, rate limit 별도 관리

### 2. Haiku vs Sonnet 비교 분석

#### 2-1. 비용 분석 (핵심)

| 항목 | Haiku 4.5 | Sonnet 4.6 | 차이 |
|------|-----------|------------|------|
| Input (백만 토큰당) | $1.00 | $3.00 | 3배 |
| Output (백만 토큰당) | $5.00 | $15.00 | 3배 |
| 블로그 제목 1회 예상 비용* | ~$0.0015 | ~$0.0045 | 3배 |
| 후킹문구 1회 예상 비용* | ~$0.0012 | ~$0.0036 | 3배 |

*예상 비용 산정 근거:
- System prompt: ~500 토큰 (input)
- User message: ~100 토큰 (input)
- 블로그 제목 출력 (24개): ~600 토큰 (output)
- 후킹문구 출력 (15개): ~450 토큰 (output)
- 합계: 블로그 제목 = 600 input + 600 output, 후킹문구 = 600 input + 450 output

**월간 비용 시뮬레이션** (일 500회 사용 가정):

| 도구 | Haiku 월비용 | Sonnet 월비용 |
|------|-------------|--------------|
| 블로그 제목 | ~$22.50 | ~$67.50 |
| 후킹문구 | ~$18.00 | ~$54.00 |
| **합계** | **~$40.50** | **~$121.50** |

두 도구 모두 무료 제공이므로 비용이 매우 중요. **Haiku는 Sonnet 대비 1/3 비용**.

#### 2-2. 한국어 품질

| 기준 | Haiku 4.5 | Sonnet 4.6 |
|------|-----------|------------|
| 자연스러운 한국어 | 양호 (threads.js에서 검증됨) | 우수 |
| 조사 처리 (은/는, 이/가) | 대체로 정확 | 정확 |
| 어색한 표현 발생률 | 간혹 발생 | 드묾 |
| 네이버 블로그 어투 | 적절 | 적절~우수 |

**핵심 판단**: 블로그 제목과 후킹문구는 모두 1줄짜리 짧은 텍스트. 이 분량에서는 Haiku와 Sonnet 간 한국어 품질 차이가 크지 않음. threads.js가 300자 수준의 글도 Haiku로 잘 처리 중이므로, 1줄 제목/문구는 충분히 Haiku로 가능.

#### 2-3. 규칙 준수 능력

| 기준 | Haiku 4.5 | Sonnet 4.6 |
|------|-----------|------------|
| 패턴별 구조 준수 | 양호 | 우수 |
| 출력 포맷 준수 (JSON/구분자) | 양호 | 우수 |
| 금지 표현 회피 | 양호 | 우수 |
| 글자수 제한 준수 | 보통 (서버 후처리 필요) | 양호 |

**핵심 판단**: 두 도구 모두 명확한 규칙(12패턴, 10가지 심리 공식)을 프롬프트로 제시하므로 구조화된 프롬프트 설계가 중요. Haiku도 threads.js에서 유형별 구조(정보형/공감형/반전형/궁금증형) + 말투 가이드를 잘 따르고 있음. 단, **출력 포맷이 복잡할수록 Sonnet이 유리**하므로, 프롬프트에서 출력 형식을 최대한 단순화하는 것이 Haiku 성공의 열쇠.

#### 2-4. 창의성

| 기준 | Haiku 4.5 | Sonnet 4.6 |
|------|-----------|------------|
| 다양한 표현 생성 | 양호 | 우수 |
| 동일 프롬프트 반복 시 변화 | 적절 | 풍부 |
| 클리셰 회피 | 보통 | 양호 |
| 키워드 맥락 연결 | 양호 | 우수 |

**핵심 판단**: 현재 JS 템플릿은 항상 고정 결과를 반환하므로, AI 전환만으로도 창의성이 대폭 향상됨. 이 관점에서 Haiku도 현재 대비 크게 개선. 프롬프트에 temperature 설정(0.8~0.9)을 활용하면 Haiku에서도 충분한 다양성 확보 가능.

#### 2-5. 속도

| 기준 | Haiku 4.5 | Sonnet 4.6 |
|------|-----------|------------|
| 평균 응답 시간 (짧은 출력) | ~1~2초 | ~2~4초 |
| 사용자 체감 | 즉시~빠름 | 약간의 대기 |

**핵심 판단**: 현재 JS 템플릿은 즉시 생성(0ms). API 전환 시 불가피하게 지연이 발생하므로, 가능한 한 빠른 모델이 UX에 유리. **Haiku가 Sonnet 대비 약 2배 빠름**.

#### 2-6. 문맥 연결

| 기준 | Haiku 4.5 | Sonnet 4.6 |
|------|-----------|------------|
| 키워드+업종 시너지 | 양호 | 우수 |
| 의미적으로 자연스러운 삽입 | 양호 | 우수 |
| 맥락에 맞지 않는 제목 생성 | 간혹 | 드묾 |

**핵심 판단**: 현재 JS 템플릿의 가장 큰 약점이 문맥 무시 삽입. AI 전환 시 두 모델 모두 크게 개선됨. Haiku도 업종+키워드 조합의 문맥을 이해하여 자연스러운 제목/문구 생성 가능.

#### 2-7. 종합 비교 및 결론

| 평가 항목 | 가중치 | Haiku 점수 | Sonnet 점수 |
|----------|--------|-----------|-------------|
| 비용 | **40%** | 10 | 4 |
| 속도 | **20%** | 9 | 6 |
| 한국어 품질 | 15% | 7 | 9 |
| 규칙 준수 | 10% | 7 | 9 |
| 창의성 | 10% | 7 | 9 |
| 문맥 연결 | 5% | 7 | 9 |

### >>> 최종 추천: Claude Haiku 4.5 (`claude-haiku-4-5-20251001`)

**추천 근거**:
1. **비용 3배 절감**: 무료 도구이므로 수익 없이 API 비용만 발생. Haiku는 월 ~$40 vs Sonnet 월 ~$120
2. **이미 검증됨**: threads.js가 동일 모델로 300자 수준의 글을 성공적으로 생성 중. 1줄 제목/문구는 더 쉬운 태스크
3. **속도 우위**: 현재 즉시 생성에서 API 전환 시 지연이 불가피한 만큼, 빠른 모델이 UX 유지에 유리
4. **한국어 품질 차이 미미**: 1줄 짧은 텍스트에서는 Haiku와 Sonnet 간 체감 품질 차이가 크지 않음
5. **프롬프트 최적화로 보완 가능**: 구조화된 프롬프트 + 서버 후처리로 Haiku의 약점 보완 가능

### 3. 프롬프트 초안 설계

#### 3-1. 블로그 제목 생성기 — System Prompt

```
당신은 네이버 블로그 제목 전문 카피라이터다. 클릭율(CTR)을 극대화하는 블로그 제목을 생성한다.

[절대 규칙]
1. 각 패턴별로 정확히 2개의 제목을 생성한다. 총 24개.
2. 모든 제목에 사용자 키워드를 자연스럽게 포함한다.
3. 제목 길이: 20~35자 (모바일 검색 결과에서 잘리지 않도록).
4. 한국어 조사(은/는, 이/가, 을/를, 으로/로)를 정확히 사용한다.
5. 숫자를 활용할 때는 키워드 맥락에 맞는 구체적 숫자를 쓴다.
6. 출력은 순수 텍스트만. 마크다운, 이모지, 해시태그 금지.
7. 각 제목은 줄바꿈으로 구분한다.
```

### 4. API 연동 방식 제안

#### 방식 비교

| 방식 | 장점 | 단점 |
|------|------|------|
| A. 기존 generate.js 활용 | 코드 재사용, 유지보수 간편 | 프롬프트를 프론트에서 전송 (노출 위험) |
| B. 새 API 파일 생성 (api/titles.js, api/hooks.js) | 프롬프트 서버 보호, rate limit 독립 | 파일 2개 추가, 코드 중복 |
| C. 기존 generate.js + 프롬프트 서버 내장 | 1개 파일로 통합, 프롬프트 보호 | generate.js 복잡도 증가 |

#### >>> 추천: 방식 B — 새 API 파일 생성

**추천 근거**:
1. **프롬프트 보호**: system prompt가 서버에 내장되어 프론트엔드에 노출되지 않음 (threads.js 패턴과 동일)
2. **독립적 rate limit**: 두 도구가 무료이므로, 블로그 글 생성기(generate.js)와 별도로 rate limit 관리 가능
3. **일관된 아키텍처**: threads.js와 동일한 패턴으로 코드 일관성 유지
4. **유지보수 용이**: 각 도구의 프롬프트를 독립적으로 수정 가능

#### 제안 파일 구조

```
api/
  generate.js     (기존 — 블로그 글 생성, Sonnet)
  threads.js      (기존 — 스레드 글 생성, Haiku)
  titles.js       (신규 — 블로그 제목 생성, Haiku)
  hooks.js        (신규 — 후킹문구 생성, Haiku)
```

#### API 스펙 제안

**api/titles.js**:
- 모델: `claude-haiku-4-5-20251001`
- max_tokens: 800
- temperature: 0.85
- Input: `{ keyword, category }`
- Output: `{ results: { p1: [t1, t2], p2: [t1, t2], ... p12: [t1, t2] }, remaining }`
- Rate limit: 무제한 (무료 도구) 또는 IP당 일 30회

**api/hooks.js**:
- 모델: `claude-haiku-4-5-20251001`
- max_tokens: 600
- temperature: 0.85
- Input: `{ industry, keyword }`
- Output: `{ results: [hook1, hook2, ... hook15], remaining }`
- Rate limit: 무제한 (무료 도구) 또는 IP당 일 30회

#### 서버 후처리 로직 (threads.js 패턴 차용)

1. **출력 파싱**: 줄바꿈 기준으로 split, 빈 줄 제거
2. **글자수 보정**: 제목 40자 초과 시 트리밍, 후킹문구 50자 초과 시 트리밍
3. **이모지 제거**: 프롬프트에서 금지했지만 Haiku가 간혹 생성할 수 있으므로 서버에서 제거 (옵션)
4. **개수 보장**: 24개/15개 미달 시 에러 처리 또는 기존 JS 템플릿 fallback

#### Fallback 전략

API 실패 시 기존 JS 템플릿을 fallback으로 사용하는 것을 권장:
- 프론트엔드에 기존 templates/HOOKS 로직을 유지
- API 호출 실패 시 자동으로 기존 로직으로 전환
- 사용자에게 "AI 서버가 바쁩니다. 기본 제목을 표시합니다." 메시지 표시

### 5. 구현 시 주의사항

1. **Prompt Caching 활용**: system prompt는 고정이므로 Anthropic의 prompt caching을 활용하면 비용 추가 절감 가능 (cache read는 0.1x 비용)
2. **temperature 설정**: 0.85 추천. 너무 낮으면 반복적, 너무 높으면 맥락 이탈
3. **기존 JS 코드 유지**: 즉시 삭제하지 말고 fallback + A/B 테스트용으로 보존
4. **Rate limit 설계**: 무료 도구지만 남용 방지를 위해 IP당 일 20~30회 권장
5. **응답 파싱 안정성**: Haiku 출력이 간혹 형식을 벗어날 수 있으므로 파싱 로직에 유연성 필요
